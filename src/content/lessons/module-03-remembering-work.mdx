## Learning goal

Motivate the **KV cache**, compute its size from hyperparameters, and see why long context + batching hits a **memory wall**.

## Builds on lessons 1–2

- Generation grows the past.
- Attention needs **keys** and **values** for positions it looks at.

## The wasteful approach

At every new token, recompute K and V for **all** past tokens, then attend. Correct—but repeats enormous work.

## The KV cache

After a token is processed, store its **key** and **value** vectors (per layer, per KV head). The next step **reuses** them and only computes K/V for the newest token.

That table is the **KV cache**—the main inference-memory object of this course.

## Canonical size formula

<Formula>
{`M_KV = 2 · B · L · H_kv · S · d_h · bpe

2 = keys AND values
B = batch size (concurrent sequences)
L = number of layers
H_kv = number of KV heads (≤ query heads if GQA/MQA)
S = sequence length (tokens in the cache)
d_h = head dimension
bpe = bytes per element (2 for FP16/BF16)`}

</Formula>

Live metrics implement this in `src/lib/kvMemory.ts` and must match `docs/glossary.md`.

### Worked example (glossary)

`B=1`, `L=32`, `H_kv=8`, `d_h=128`, `S=4096`, `bpe=2` → **512 MiB** of KV alone (not counting weights).

## Weights vs cache

**Weights** are large but mostly **fixed** after load. **KV** grows with `S` and `B`. The metrics panel shows a **rough** weight estimate from the model preset so you can compare magnitudes—not a vendor memory profiler.

<TryThis>

- Double **sequence length S**: KV bytes should roughly double.
- Raise **batch size B**: same idea for concurrent chats.
- Switch preset to a GQA-shaped config: fewer `H_kv` shrinks cache vs full MHA with the same query width.
- Lower **HBM budget** (via utilization / OOM): when KV exceeds the budget, the scene flags **over budget**.

</TryThis>

## The memory wall (serving sense)

Even with spare FLOPs, you cannot serve large batches of long contexts if HBM cannot hold weights **plus** growing KV (and other runtime overhead). Systems respond with paging (lesson 5), reuse (lesson 6), quantization (lesson 7), and eviction (lesson 8).
