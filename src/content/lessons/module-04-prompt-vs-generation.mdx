## Learning goal

Separate **prompt processing** from **token-by-token generation**, and connect each phase to different performance pressures.

## Builds on lesson 3

You have a KV cache that **fills** during the prompt and **grows** during the reply.

## Two phases

| Phase                  | Also called | What happens to KV                                     | User-visible metric                    |
| ---------------------- | ----------- | ------------------------------------------------------ | -------------------------------------- |
| **Reading the prompt** | Prefill     | Compute K/V for many prompt tokens; cache is populated | **TTFT** — time to first token         |
| **Writing the reply**  | Decode      | Append K/V for one (or a few) new tokens per step      | **TPOT / ITL** — time per output token |

Prefill often looks **compute-heavy** (large parallel work over the prompt). Decode often looks **memory-bandwidth-heavy** (reload weights and read a growing cache each step). These are **tendencies**, not theorems—short prompts and odd batches can flip the bottleneck.

## What the scene shows

- **Compute brick** vs **HBM tray** on a GPU node: which one “glows” depends on the **Phase** control.
- **Bandwidth ribbon**: memory traffic metaphor during decode-leaning settings.
- **Batch slots**: intuition for **iteration-level** batching—requests can enter/leave a batch as they finish (`S11` Orca), rather than waiting for the slowest sequence in a static batch.

## Why servers care

A long **prefill** can delay **decode** for other users if both share the same GPU (interference). Later systems use chunked prefills (`S13`/`S14`) or disaggregated prefill/decode pools (`S12`). You only need the conflict today; schedulers get depth in advanced modules.
