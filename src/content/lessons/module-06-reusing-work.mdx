## Learning goal

Explain how **shared prefixes** let a server reuse KV work across requests, and how a **radix / prefix tree** organizes that reuse.

## Builds on lessons 3–5

- KV stores past keys and values (lesson 3).
- Pages make physical sharing natural (lesson 5).

## The opportunity

Many requests share the same **leading text**:

- System prompts (“You are a helpful assistant…”)
- Few-shot example blocks
- Multi-turn chat history prefixes
- Template headers in agents / RAG wrappers

If two prompts share an **exact** token prefix, the engine can reuse the KV already computed for that trunk—saving **compute** (skip re-prefill of the shared part) and **memory** (one physical copy of shared pages).

## RadixAttention intuition (SGLang)

Instead of discarding cache when a request finishes, retain prefixes in a structure optimized for prefix search—commonly described as a **radix tree** of tokens (`S15`):

- **Root → shared trunk** (common system prompt)
- **Branches** for divergent user text
- **Lookup** finds the longest matching prefix
- **Insert / evict** manage what stays resident (LRU-like policies in real systems)

vLLM-class stacks may use **block hashing** for automatic prefix caching (`S17`)—same family of idea, different index.

## When reuse fails

Reuse requires **exact** token identity (and consistent model/adapter/position handling as implemented). Divergence at token 1 means **no** shared trunk—even if English “looks similar.”
