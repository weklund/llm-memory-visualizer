## The usual picture is wrong

You already know the product: you type a message, a language model writes a reply. That is enough background.

The natural picture in your head is often this: _the model “understands” the question, forms the whole answer, then dumps a paragraph._ That picture is useful for using chat apps. It is a poor picture of how the reply is actually produced.

<MentalModelContrast />

What actually happens is narrower and more mechanical. The model repeatedly asks a smaller question:

> Given everything so far, what is a good **next small piece** of text?

Those pieces are **tokens**—sometimes whole words, sometimes fragments, sometimes punctuation. For this lesson, treat them as beads on a string. You do **not** need tokenization details.

## One full model pass, one piece, then again

Important nuance: this is **not** “the model casually picks the next word.” Each next token is the output of a **full pass through the whole model**—then that token is appended, and the **whole model runs again** for the token after that.

Engineers describe generation as a loop:

1. Look at the text so far (your prompt, then anything already written).
2. Run the **entire** model once → get **one** next token.
3. **Append** that token to the text so far.
4. Repeat—another full pass—until the reply stops.

A ten-token reply is roughly **ten full model runs** in a row (plus the work to process the prompt). Each step conditions on a **growing past**. That process is called **autoregressive generation**—the name is optional; the loop of full passes is not.

## The past in play keeps growing

The same loop explains three everyday situations:

| What you do                         | What grows for the next token                   |
| ----------------------------------- | ----------------------------------------------- |
| You send a **longer prompt**        | More tokens exist before the first output token |
| The model writes a **longer reply** | Each new token joins the past for the next step |
| You continue a **multi-turn chat**  | Earlier turns usually stay in context           |

Nothing mystical: the loop always asks “what next, given so far?”—and “so far” is simply longer. That is why systems care about **sequence length** long before we talk about GPUs.
