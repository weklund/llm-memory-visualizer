## Learning goal

Explain how an LLM writes text **one token at a time**, and why the amount of “past text in play” grows during a reply.

## Start here

You do **not** need transformer math. If you know chat models read a prompt and write a reply, you are ready.

## The generation loop

An LLM does not compose a full answer in one mysterious step. It repeatedly asks:

> Given everything so far, what is a good **next token**?

A **token** is a chunk of text the model uses as a unit (a whole word, part of a word, or punctuation). Roughly:

1. Start with your **prompt** (plus any system instructions).
2. Predict the next token.
3. **Append** that token to the context.
4. Repeat until the reply finishes or a length limit is hit.

That process is **autoregressive generation**: each step conditions on a **growing past**.

## Why length grows

| What you do             | What happens to past text                                         |
| ----------------------- | ----------------------------------------------------------------- |
| Send a longer prompt    | More tokens exist before the first output token                   |
| Generate a longer reply | Each new token adds to the past for the next step                 |
| Multi-turn chat         | Earlier turns remain in context unless the product truncates them |

The purple beads in the scene are a **display-sized** token stream (not one mesh per real token). Drag **Generation progress** to move the “next” cursor; stretch **How long is the past?** to feel a longer history.

<TryThis>

- Set progress near 0%: mostly empty / dim future slots.
- Set progress near 100%: the stream is fully “written.”
- Increase past length: more beads appear (capped for performance).

</TryThis>

## Two meanings of “memory” (important split)

1. **Product language** — “the model remembers our conversation” (text kept in the context window).
2. **Engine language** — structures the GPU stores so it does not redo work (starts in lesson 3 as the **KV cache**).

This lesson only needs (1). Lesson 2 asks _how_ the model looks at past tokens. Lesson 3 asks _what_ the engine stores for that looking-back.
