## Learning goal

State the **efficiency vs privacy** tension of shared prefix caches, describe **timing / scheduling side channels** at a high level, and list **mitigation directions**—without an exploit playbook.

## Builds on lesson 6

Cross-request KV reuse is valuable. In **multi-tenant** serving, that shared state can become an **oracle**: “was this prefix already cached?”

## Threat model (conceptual)

| Piece           | Teaching meaning                                                                                           |
| --------------- | ---------------------------------------------------------------------------------------------------------- |
| **Shared pool** | Tenants A and B may hit the same physical prefix pages                                                     |
| **Observable**  | Latency (e.g. TTFT), queue order, or other API-visible timing—not necessarily raw K/V download             |
| **Probe**       | Attacker-controlled prompts that test whether a candidate prefix is hot                                    |
| **Risk**        | Partial reconstruction of another user’s prompt or template in researched attacks (`S21` PROMPTPEEK-class) |

This lesson teaches **why the channel exists** and **what defenders change**. It does **not** walk through operational attack steps.

## Efficiency vs isolation

| Mode               | Upside                              | Downside                      |
| ------------------ | ----------------------------------- | ----------------------------- |
| **Shared cache**   | Higher hit rate, lower cost/latency | Cross-tenant signals          |
| **Isolated cache** | Smaller blast radius                | Duplicate storage, less reuse |

Real mitigations also include rate limits, noise, monitoring, reduced cross-user matching, and careful scheduling—research is active (`S22`).

## Defender checklist (non-exhaustive)

1. **Isolate** by trust domain (user, org, API key).
2. **Limit** who can share with whom (selective sharing).
3. **Detect** abusive probing (rate / pattern).
4. **Reduce** API-visible timing differences where feasible.
5. **Assume** shared caches are a security boundary decision, not a free lunch.
